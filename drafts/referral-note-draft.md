## Referral Note (Clinical Notes data class) — Comment Position Summary

**Element definition:** A narrative summary requesting an opinion, advice, or service from a clinician for a patient. Identified by LOINC 57133-1 (Referral note). Designed to pair with the new Referral Order element in the Orders data class to provide both structured request and narrative context.

| Position | Organizations | Reasons |
|---|---|---|
| **OPPOSE** | AHA (hospital trade association), FAH (hospital trade association) | Doesn't reflect existing workflows — referrals initiated via orders, documented in progress notes; standalone element adds burden and redundancy |
| **OPPOSE / REDESIGN** | *(none)* | — |
| **MIXED / OPPOSE** | EHR Association (vendor trade association) | Standalone note duplicative of Referral Order; scope as attachment to the order |
| **SUPPORT / with CHANGES** | AMA (physician professional society), Epic (EHR vendor), Oracle Health (EHR vendor), WEDI (admin trade association) | AMA: clarify links to Appointment/Encounter. Epic: distinguish from existing Reason for Referral. Oracle: same LOINC for note and CDA document type — disambiguate via encoding with HL7. WEDI: distinguish referral/consult/transition of care |
| **SUPPORT** | Regenstrief Institute (research/informatics) | LOINC 57133-1 specified; pairs well with Referral Order |

**Comments grouped by position:**

**Strongly supportive**
- **Regenstrief Institute** [research/informatics institute] — Supports inclusion. The addition of Referral Note (LOINC 57133-1) to Clinical Notes is "a natural complement to the new Referral Order element in the Orders class." The pairing enables both the structured order and the narrative context to be exchanged. Frames the order + note design as well-designed for referral workflows; aligned with care coordination reporting.

**Supportive with refinements**
- **American Medical Association / AMA** [physician professional society] — Lists Referral Note alongside Medical Device Order, Nutrition Order, and Referral Order under "Orders and Care Coordination." Supports additions that improve care coordination, reduce administrative burden, and support transitions of care — particularly for referrals and device ordering. To avoid fragmentation, recommends ONC clarify how Referral Order and Referral Note relate to existing referral/consult workflows and how they are linked to scheduling (Appointment) and clinical interactions (Encounter).
- **Epic** [EHR vendor] — Supports the addition. Encourages ONC to clarify the distinction between the Referral Note data element and the **existing Reason for Referral data element**. As currently defined, both elements appear to describe the clinical justification for a referral, and without clearer boundaries between them, implementers may exchange duplicative information or interpret the elements inconsistently.
- **Oracle Health** [EHR vendor] — Supports inclusion of the proposed Referral Note narrative data element. Agrees with the Clinical Notes data class usage note that this and other Clinical Notes "should not be confused as representative of the CDA C-CDA document types." However, raises a specific encoding concern: **the same LOINC code is used for both the narrative note and the document type**, so the elements will continue to be interpreted as the same despite ONC's intent. Urges ONC to work with HL7 to fully disambiguate these two concepts so they can be recognized as different based on appropriate encoding.
- **WEDI** [administrative/transactions trade association] — Cites Referral Order and Referral Note among v7 additions that "could materially reduce administrative burden and improve interoperability." Three recommendations on the pair: (1) clarify how Referral Order and Referral Note connect to Appointment, Encounter, and the receiving organization; (2) provide guidance and examples distinguishing "referral request" vs. "consult request" vs. "transition of care" communications to reduce inconsistent implementations; (3) encourage alignment with API-based coordination workflows so referral content can be used downstream — "referrals are a major friction point across providers and payers."

**Mixed / oppose**
- **EHR Association** [vendor trade association] — Supports the inclusion of additional referral-related content within USCDI but raises questions about the relationship between Referral Note and the existing Referral Order data class. As currently defined, **a standalone Referral Note may be duplicative of the referral order itself**. Unclear whether the element is intended to represent the clinical rationale accompanying a referral or a distinct clinical document separate from the referral order. In practice, many systems do not maintain a separate referral note but instead include relevant clinical context directly within the referral order or as part of a general clinical note. Recommends ONC clarify the distinction between Referral Note and Referral Order, including whether a Referral Note must be associated with a structured Referral Order. **If the intent is to capture the clinical narrative supporting a referral, the EHR Association suggests scoping this element as an attachment or addendum to the referral order rather than as a standalone note**, to better align with real-world system design and workflows.

**Oppose**
- **American Hospital Association / AHA** [hospital trade association] — Urges ONC not to adopt the proposed Referral Note data element. The element entails a narrative summary that may include the reason for referral, relevant patient history, and requested services or questions — but **AHA argues this data element does not reflect existing clinical workflows**. Typically, referrals are initiated through an order, and documentation is completed within clinical progress notes. The addition of a referral note data element outside the progress note would introduce administrative burden and redundancies without commensurate improvement in coordination. AHA places Referral Note in the same opposition tier as Adverse Events — "elements that conflict with existing processes and where risks outweigh benefits."
- **Federation of American Hospitals / FAH** [hospital trade association] — Has concerns about including Referral Note as a distinct USCDI data element. Same workflow argument as AHA: **referrals are typically initiated through orders and documented within existing progress notes**. Requiring a discrete referral note solely for interoperability purposes would introduce unnecessary operational burden without improving care coordination. FAH places Referral Note among elements requiring "foundational standards work" before advancement.

**Summary of Comments:** Referral Note is the only v7 element reviewed so far where the substantive commentary comes entirely from organizational PDF letters — there are zero ISP form-based comments on the element, an unusual pattern that suggests the topic surfaced primarily through deliberative organizational review rather than individual stakeholder advocacy. The disagreement is structural: every commenter agrees that the **content** of a referral note (clinical rationale, history, requested services) is valuable for care coordination, but they disagree on whether that content needs its own USCDI data element. **The hospital trade associations** (AHA, FAH) [hospital trade associations] take the strongest opposition position with identical reasoning: in real clinical practice, referrals are initiated through orders and the supporting clinical narrative is documented in progress notes — so a standalone Referral Note element duplicates documentation that already exists in two other USCDI structures (Referral Order plus Clinical Progress Notes) and adds administrative burden without improving coordination. **The vendor trade association** (EHR Association) [vendor trade association] makes a softer version of the same argument: many EHR systems don't maintain a separate referral note, instead including clinical context within the referral order itself or in general clinical notes. Their proposed compromise — scope Referral Note as an attachment or addendum to Referral Order rather than a standalone element — would partially address the hospital concern by anchoring the note to its parent order. **The supporters with refinements** raise three distinct boundary-clarification concerns. **Epic** [EHR vendor] points to the existing **Reason for Referral** element and asks how Referral Note differs — both currently appear to capture clinical justification for a referral. **Oracle Health** [EHR vendor] points to a deeper encoding problem: USCDI's Clinical Notes class explicitly states these aren't C-CDA document types, but **Referral Note and the C-CDA referral document type share the same LOINC code (57133-1)** — so without further disambiguation through encoding, implementers will treat them as the same thing in practice. Oracle's ask is that ONC work with HL7 to resolve this through encoding. **AMA** [physician professional society] and **WEDI** [administrative trade association] both ask ONC to clarify how Referral Note connects to **Appointment, Encounter, and the receiving organization**, with WEDI specifically asking for examples distinguishing referral request, consult request, and transition of care — the three workflow patterns that all currently get blurred under "referral." Only **Regenstrief Institute** [research/informatics institute] supports the element without qualifications, framing the order + note pairing as well-designed for referral workflows. The most striking pattern in the comment record: TMA [state physician society] did not list Referral Note among their 15 "defer until vocabulary" elements, which is unusual — Referral Note is one of the few v7 additions where TMA effectively offered no objection — possibly because LOINC 57133-1 is already specified in the draft.

**Notably absent:** AHIP, AMIA, ANI, AQIPS, NCQA (notable given their general support for referral-related additions), **HL7** (especially notable given Oracle's ask is for ONC to work with HL7 to disambiguate the LOINC encoding issue), MEDITECH, AAAAI, FDA, CDC, APHL, CSTE, AHIMA, ACLA, SHIELD, Vizient, Wolters Kluwer, Allina Health, Emory Healthcare, UI Health, Providence Health, CARIN Alliance, PACIO Project, Vega Health, SNOMED International, NCPDP, Altarum Institute, and individual ISP commenters are all silent on Referral Note. The absence of HL7 is the single most consequential gap given that Oracle Health's primary technical concern (the shared LOINC code between narrative note and CDA document type) requires HL7 to resolve. The complete absence of patient advocacy and consumer groups, despite referrals being a frequent patient-experience friction point, is also notable. The lack of ISP form comments is unique to this element among the v7 additions reviewed so far — every other element has had at least one or two individual or organizational ISP submissions.
