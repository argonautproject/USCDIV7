
# CLAUDE.md

## Repository purpose
This repo is a working area for authoring new US Core profiles aligned
to USCDI v7. Output is FHIR StructureDefinition JSON intended to be
contributed upstream to HL7/US-Core. This is published as a github page to share
during development so supports a full ig-publication stack but is not formally published
as an HL7 guide.

All builds use `/Users/ehaas/Documents/FHIR/US-Core/publish.sh` and run in two steps:
  1. `/Users/ehaas/Documents/FHIR/US-Core/publish.sh -sy` — run SUSHI and convert YAML sources to JSON
  2. `/Users/ehaas/Documents/FHIR/US-Core/publish.sh -iq` — run the IG Publisher and view the QA report
The IG Publisher JAR location is configured on line 43 (`ig_path`) of publish.sh.
After a successful build, the IG publisher generates a detailed QA page at `output/qa-eslintcompact.txt` — it is NOT committed to git
- `.qa_previous.txt` is a local baseline copy saved before each build — also NOT committed
- When asked about QA, sort and compare these two files and only list new and resolved issues


## FHIR version
- FHIR R4 (4.0.1)
- US Core baseline: current main
- USCDI version: v7 (draft version published January 2027)

### Key Directories

| Directory | Purpose |
|-----------|---------|
| `input/resources-yaml/` | YAML source for profiles, extensions, search parameters, capability statements |
| `input/examples-yaml/` | YAML source for example resources |
| `input/pagecontent/` | Markdown narrative pages (index.md, conformance.md, guidance.md, etc.) |
| `input/intro-notes/` | Per-artifact intro/notes markdown fragments (144 files) |
| `input/includes/` | Reusable Jekyll include fragments |
| `input/data/` | CSV/YAML data collections used by Jekyll Liquid templates |
| `input/liquid/` | FHIR Liquid templates for generating narratives |
| `input/images/` | Images and downloadable tables |
| `fsh-generated/` | SUSHI output (ImplementationGuide resource, menu.xml) |
| `output/` | Built IG (HTML, JSON, XML, packages) — gitignored |

### Configuration Files

- **`sushi-config.yaml`** — Main IG configuration (version, dependencies, page hierarchy, parameters). SUSHI only generates the ImplementationGuide resource from this; it does NOT author FHIR artifacts.
- **`ig.ini`** — Points IG Publisher to the generated ImplementationGuide resource and specifies the HL7 template version.
- **`publication-request.json`** — HL7 publication metadata (updated before each publication).
- **`input/ignoreWarnings.txt`** — QA report warning suppressions.


## Naming Conventions

- Profiles: `StructureDefinition-us-core-[resource].json`
- Extensions: `StructureDefinition-us-core-[name].json`
- Search Parameters: `SearchParameter-us-core-[resource]-[param].json`
- Intro/Notes files: `StructureDefinition-us-core-[resource]-intro.md` / `-notes.md`

## Authoring conventions
- Differential-only authoring; do not hand-edit the snapshot. Snapshots
  are generated by the publisher.

## Jekyll/Liquid Templating

Pages use Jekyll with Liquid templating extensively:
- **Include files** (`input/includes/`): Reusable fragments inserted via `{% include filename.md %}`. Include files with parameters document their parameters in inline comments at the top.


## USCDI alignment
Each profile maps to one or more USCDI v7 Data Classes / Data Elements.
USCDI data Class and descriptions are in /Users/ehaas/Documents/FHIR/US-Core/my-notes/USCDIV7/USCDI_Draft_Version7.pdf
When generating descriptions, cite the Data Class name
exactly as it appears in USCDI v7, not paraphrased.


## What Claude should and shouldn't touch
- OK to edit: files in `input/`
- OK to read for reference: `output/`, `temp/`, `template/`
- Do not edit: `output/`, `temp/`, `template/`, `input/resources` , `input/examples `(these are generated files)
- Do not invent: bindings, mustSupport flags, USCDI mappings, extensions or history not present in the source SD. If a fact is needed and not
  in the file, ask rather than fabricate.

## House style for prose (READMEs, descriptions, commit messages)
- Present tense, third person. No "we," "you."
- RFC 2119 keywords (SHALL/SHOULD/MAY) used in the normative sense only.
  - Use "might" or "can" instead of "MAY"
- No marketing adjectives: "robust," "comprehensive," "seamless."
- No hedges: "typically," "generally," "may currently."
- Plain ASCII; straight quotes; no em dashes.
- Spell out acronyms on first use except FHIR, US Core, USCDI, HL7, SNOMED CT, LOINC.
